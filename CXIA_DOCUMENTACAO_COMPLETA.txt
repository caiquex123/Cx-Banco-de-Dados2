1.1. O QUE É O SISTEMA CXIA DB?

O CxIA DB é uma implementação completa de backend "self-hosted" (auto-hospedado) 
desenvolvida em Python puro, utilizando o framework FastAPI e o banco de dados 
SQLite. Ele foi projetado especificamente para alimentar aplicações SaaS (Software 
as a Service) que necessitam de persistência de dados real, sem depender de 
serviços de nuvem pagos ou complexos.

Diferente de soluções tradicionais que exigem servidores PostgreSQL dedicados, 
configurações de Docker complexas ou assinaturas mensais (como Supabase, Firebase 
ou AWS), o CxIA DB roda diretamente no sistema operacional do administrador, 
utilizando um único arquivo (.db) para armazenar todos os dados da aplicação.

1.2. PROBLEMA RESOLVIDO

Aplicações modernas frequentemente enfrentam o dilema da persistência de dados:
- Opção A: Usar localStorage no navegador.
  * Problema: Os dados somem se o usuário limpar o cache, trocar de dispositivo 
    ou usar aba anônima. Não há sincronização entre dispositivos.
  
- Opção B: Usar serviços de nuvem (Supabase, Firebase).
  * Problema: Custos recorrentes que escalam com o uso, dependência de internet 
    estável para configuração, vendor lock-in (dificuldade de migrar depois) e 
    preocupações com privacidade de dados (dados hospedados em servidores de 
    terceiros).

O CxIA DB resolve isso oferecendo uma "Nuvem Pessoal". Você, administrador, 
transforma seu próprio computador (ou um servidor dedicado simples) no provedor 
de dados. O frontend da aplicação (hospedado na Vercel, Netlify, etc.) conecta-se 
via URL configurável ao seu backend local.

1.3. FILOSOFIA DE DESIGN: "SOBERANIA DE DADOS"

O sistema foi construído sobre três pilares fundamentais:

1. Zero Custo de Infraestrutura:
   Utiliza apenas software livre e nativo. Não há mensalidades de banco de dados.
   O único custo é a eletricidade do equipamento onde o backend está rodando.

2. Controle Total:
   O arquivo `cxia.db` pertence a você. Ninguém mais tem acesso aos dados dos 
   seus usuários a menos que você conceda. Você pode fazer backup copiando um 
   único arquivo. Você pode inspecionar as tabelas com qualquer visualizador SQLite.

3. Simplicidade Radical:
   Sem Docker, sem Kubernetes, sem variáveis de ambiente complexas de cloud.
   Para instalar: instale Python, rode `pip install -r requirements.txt` e execute 
   `python main.py`. Para fazer backup: copie o arquivo `.db`.

2.2. STACK TECNOLÓGICO

- Linguagem Backend: Python 3.9+
- Framework Web: FastAPI (Alta performance, assíncrono, geração automática de docs)
- Banco de Dados: SQLite 3 (Nativo do Python, sem instalação extra)
- Autenticação: PyJWT (JSON Web Tokens)
- Hashing: hashlib (SHA-256) + hmac
- Servidor ASGI: Uvicorn (Embutido no FastAPI)
- Frontend: React + TypeScript + Vite
- Cliente HTTP: Fetch API nativa do navegador

2.3. TOPOLOGIA DE REDE

O sistema opera em um modelo Cliente-Servidor tradicional, mas com uma twist de 
implantação flexível:

Cenário A: Desenvolvimento Local
- Frontend: http://localhost:5173
- Backend: http://localhost:8000
- Comunicação: Loopback local (rápido, sem latência).

Cenário B: Produção Híbrida (Recomendado)
- Frontend: https://meu-saas.vercel.app (Hospedado na Vercel)
- Backend: http://SEU_IP_PUBLICO:8000 (Seu PC com IP Fixo ou DDNS)
- Segurança: Requer túnel HTTPS (ex: Cloudflare Tunnel, Ngrok) ou configuração 
  correta de firewall e SSL reverso (Nginx/Caddy).

2.4. MODELO DE IMPLANTAÇÃO (DEPLOYMENT)

O backend é desenhado para ser "Stateless" em relação à lógica, mas "Stateful" 
em relação aos dados (que ficam no disco).

1. O script `main.py` inicializa o servidor Uvicorn.
2. Ao receber requisições, o FastAPI roteia para os controllers apropriados.
3. Os controllers abrem conexões com o SQLite usando um pool gerenciado.
4. As operações de leitura/escrita são feitas com transações explícitas.
5. A resposta é serializada em JSON e enviada de volta ao cliente.

O modo WAL (Write-Ahead Logging) do SQLite é ativado por padrão, permitindo 
múltiplas leituras concorrentes enquanto uma escrita ocorre, essencial para 
ambientes web com múltiplos usuários simultâneos.

3.2. DESCRIÇÃO DETALHADA DOS ARQUIVOS PRINCIPAIS

- main.py:
  Responsável por instanciar a aplicação FastAPI, configurar o middleware de CORS 
  (permitindo que o frontend em outro domínio acesse a API), incluir os routers 
  e iniciar o servidor Uvicorn. Também define a rota `/health` para monitoramento.

- database.py:
  Implementa o padrão de gerenciamento de conexão. Usa `contextlib` para garantir 
  que as conexões sejam fechadas corretamente após o uso. Aplica configurações 
  críticas de performance e integridade no SQLite (`journal_mode=WAL`, 
  `foreign_keys=ON`).

- models.py:
  Contém o schema do banco de dados em formato SQL bruto. É executado na inicialização 
  para garantir que todas as tabelas existam (`CREATE TABLE IF NOT EXISTS`). 
  Define as chaves primárias, estrangeiras e restrições de verificação (CHECK).

- schemas.py:
  Define a estrutura de dados esperada nas requisições HTTP usando Pydantic. 
  Garante que o frontend envie os campos corretos (ex: email válido, senha mínima) 
  antes mesmo da lógica de negócio ser executada. Retorna erros 422 automáticos 
  se a validação falhar.

4. MODELAGEM DE DADOS (SCHEMA)
================================================================================

4.1. VISÃO GERAL

O banco de dados é relacional, consistindo em 5 tabelas principais interligadas 
por chaves estrangeiras. O design foca na integridade dos dados e na eficiência 
de consultas frequentes (listar conversas, buscar mensagens).

4.2. TABELA: usuarios

Armazena a identidade digital dos usuários.

Campos:
- id (TEXT, PK): UUID gerado no backend. Único e imutável.
- nome (TEXT): Nome de exibição do usuário.
- email (TEXT, UNIQUE): Identificador único para login. Indexado.
- senha_hash (TEXT): Hash da senha. Nunca armazenamos senha em texto puro.
- photo_url (TEXT): URL opcional para avatar.
- provider (TEXT): Origem da conta ('email', 'google', 'guest').
- is_guest (INTEGER): 1 se for conta temporária, 0 se permanente.
- ativo (INTEGER): Soft delete. 0 desativa o usuário sem apagar dados.
- criado_em (TEXT): Timestamp ISO 8601 da criação.
- atualizado_em (TEXT): Timestamp da última atualização.

4.3. TABELA: conversas

Representa um tópico ou sessão de chat entre o usuário e a IA.

Campos:
- id (TEXT, PK): UUID da conversa.
- user_id (TEXT, FK): Referência a `usuarios.id`. ON DELETE CASCADE (se o usuário 
  for apagado, suas conversas somem).
- titulo (TEXT): Título gerado automaticamente ou pelo usuário.
- is_private (INTEGER): 1 se a conversa não deve aparecer em listas públicas 
  (futuro recurso).
- criado_em (TEXT): Data de início.
- atualizado_em (TEXT): Data da última mensagem. Usado para ordenação.

4.4. TABELA: mensagens

O conteúdo histórico do chat.

Campos:
- id (TEXT, PK): UUID da mensagem.
- conversa_id (TEXT, FK): Referência a `conversas.id`. ON DELETE CASCADE.
- role (TEXT): Quem falou? ('user', 'assistant', 'system'). Restrição CHECK.
- content (TEXT): O texto da mensagem. Pode ser longo.
- thinking_time (REAL): Tempo que a IA levou para responder (em segundos).
- criado_em (TEXT): Timestamp exato da mensagem.

4.5. TABELA: tokens_usuarios

Gerencia a cota de uso da IA (sistema de limitação).

Campos:
- id (TEXT, PK): UUID do registro de tokens.
- user_id (TEXT, FK): Referência única a `usuarios.id`.
- tokens_usados (INTEGER): Acumulado gasto na janela atual.
- limite_tokens (INTEGER): Teto máximo permitido (ex: 50.000 para Free).
- janela_inicio (TEXT): Marco inicial da contagem de 8 horas.
- plano (TEXT): 'free' ou 'pro'. Define o limite.
- atualizado_em (TEXT): Última vez que o saldo foi tocado.

4.6. TABELA: sessoes

Gerencia o estado de "logado" via JWT.

Campos:
- id (TEXT, PK): UUID da sessão.
- user_id (TEXT, FK): Dono da sessão.
- token (TEXT, UNIQUE): O JWT string. Indexado para busca rápida.
- expires_at (TEXT): Quando a sessão expira.
- criado_em (TEXT): Data de login.

4.7. ÍNDICES E PERFORMANCE

Para garantir velocidade mesmo com milhares de registros, os seguintes índices 
são criados automaticamente:

- idx_conversas_user_id: Acelera `SELECT * FROM conversas WHERE user_id = ?`.
- idx_mensagens_conversa_id: Acelera carregamento do chat.
- idx_tokens_user_id: Acesso rápido ao saldo.
- idx_sessoes_token: Validação instantânea do JWT na DB.
- idx_sessoes_expires_at: Limpeza eficiente de sessões expiradas.

4.8. INTEGRIDADE REFERENCIAL

O SQLite é configurado com `PRAGMA foreign_keys = ON`. Isso garante que:
1. Você não pode criar uma mensagem para uma conversa que não existe.
2. Se um usuário é deletado, TODAS as suas conversas, mensagens, tokens e sessões 
   são apagados automaticamente (CASCADE), evitando dados órfãos e corrupção.

5.1. PADRÕES DE COMUNICAÇÃO

  Descrição: Autentica usuário existente.

  Descrição: Cria conta temporária sem senha.

  Descrição: Invalida o token atual.
  Auth: Obrigatória.
  Retorno (200): { "message": "Logout realizado" }

GET /conversas
  Descrição: Lista histórico de conversas do usuário.
  Query Params: ?limit=20&offset=0 (Paginação futura)
  Retorno (200): { "conversas": [ {...}, {...} ] }

POST /conversas
  Descrição: Inicia novo chat.
  Body: { "titulo": "Projeto X" }
  Retorno (201): { "id": "...", "titulo": "...", "criado_em": "..." }

PUT /conversas/{id}
  Descrição: Renomeia ou arquiva conversa.
  Body: { "titulo": "Novo Título", "is_private": true }
  Retorno (200): Conversa atualizada.

DELETE /conversas/{id}
  Descrição: Remove conversa e todas as mensagens.
  Retorno (200): { "message": "Deletado" }

5.4. ENDPOINTS DE MENSAGENS (/mensagens)

GET /mensagens/{conversa_id}
  Descrição: Carrega histórico de um chat específico.
  Retorno (200): { "mensagens": [ {"role": "user", "content": "..."}, ... ] }

POST /mensagens
  Descrição: Adiciona mensagem ao chat.
  Body: { "conversa_id": "...", "role": "user", "content": "Olá", "thinking_time": 1.5 }
  Retorno (201): Mensagem criada.
  Nota: Atualiza automaticamente `atualizado_em` da conversa pai.

DELETE /mensagens/{id}
  Descrição: Apaga mensagem específica.
  Retorno (200): { "message": "Deletado" }

5.5. ENDPOINTS DE TOKENS (/tokens)

GET /tokens/saldo
  Descrição: Verifica quota disponível.
  Retorno (200): { "tokens_usados": 1200, "limite_tokens": 50000, "plano": "free" }

POST /tokens/consumir
  Descrição: Debita tokens por uso de IA.
  Body: { "amount": 150, "descricao": "Geração de resposta" }
  Lógica: Verifica janela de 8h. Se expirada, reseta contador. Se saldo insuficiente, nega.
  Retorno (200): { "allowed": true, "tokens_restantes": 48650 }
  Erros: 402 (Payment Required / Saldo Insuficiente).

POST /tokens/upgrade
  Descrição: Migra de Free para Pro.
  Body: { "novo_plano": "pro" }
  Retorno (200): { "plano": "pro", "limite_tokens": 500000 }

5.6. ENDPOINTS DE SISTEMA

GET /health
  Descrição: Check de vitalidade do servidor.
  Retorno (200): { "status": "ok", "version": "1.0", "db_size": "1024KB" }

5.7. CÓDIGOS DE STATUS HTTP

- 200 OK: Sucesso na operação.
- 201 Created: Recurso criado com sucesso (Registro, Nova Conversa).
- 400 Bad Request: Dados inválidos ou regra de negócio violada.
- 401 Unauthorized: Token ausente, inválido ou expirado.
- 402 Payment Required: Saldo de tokens insuficiente.
- 403 Forbidden: Ação não permitida (ex: tentar editar conversa de outro usuário).
- 404 Not Found: Recurso não encontrado.
- 500 Internal Server Error: Erro crítico no servidor ou banco de dados.

================================================================================
6. SEGURANÇA E AUTENTICAÇÃO
================================================================================

6.1. HASHING DE SENHAS

Embora o código utilize SHA-256 para simplicidade e zero dependências externas 
pesadas, a recomendação para produção de alta segurança é usar `bcrypt`.
No implementation atual:
- A senha recebida é concatenada com um salt (pode ser implementado) e hashada.
- `hashlib.sha256(senha.encode()).hexdigest()`
- Comparação segura usando `hmac.compare_digest()` para evitar ataques de timing.

6.2. JSON WEB TOKENS (JWT)

O sistema usa JWT stateless para autenticação.
Estrutura do Token:
- Header: { "alg": "HS256", "typ": "JWT" }
- Payload: { "sub": "user_id", "exp": timestamp_expiracao, "iat": timestamp_emitido }
- Signature: Assinado com `JWT_SECRET` (variável de ambiente crítica).

Validade: 30 dias.
Ao fazer login, o servidor gera o token e o frontend o salva no localStorage.
Em cada requisição protegida, o frontend envia: `Authorization: Bearer <token>`.
O middleware do FastAPI decodifica o token, verifica a assinatura e a expiração. 
Se válido, injeta os dados do usuário na rota (`current_user`).

6.3. GESTÃO DE SESSÕES

Apesar do JWT ser stateless, a tabela `sessoes` mantém um registro dos tokens ativos.
Isso permite:
- Logout efetivo: O token é removido da tabela (ou listado em blacklist).
- Revogação: Se um usuário tiver a conta comprometida, o admin pode limpar a tabela 
  `sessoes` daquele usuário, forçando re-login em todos os dispositivos.

6.4. CORS (CROSS-ORIGIN RESOURCE SHARING)

Como o frontend (ex: vercel.app) e o backend (ex: meu-ip:8000) estão em domínios 
diferentes, o navegador bloquearia as requisições sem CORS.
Configuração no `main.py`:
```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://meu-saas.vercel.app", "http://localhost:5173"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```
IMPORTANTE: Em produção, nunca use `allow_origins=["*"]` se estiver lidando com 
dados sensíveis e autenticação. Liste explicitamente os domínios permitidos.

6.5. PREVENÇÃO CONTRA SQL INJECTION

Todas as consultas ao banco usam parâmetros vinculados (bound parameters).
ERRADO (Vulnerável):
`cursor.execute(f"SELECT * FROM usuarios WHERE email = '{email}'")`

CERTO (Seguro):
`cursor.execute("SELECT * FROM usuarios WHERE email = ?", (email,))`

O SQLite trata os parâmetros como dados, não como código SQL executável, 
neutralizando tentativas de injeção.

6.6. PROTEÇÃO CONTRA RACE CONDITIONS

No consumo de tokens, dois pedidos simultâneos poderiam ler o mesmo saldo, 
ambos aprovarem o gasto e debitar apenas uma vez, gerando prejuízo.
Solução: Transações com `BEGIN IMMEDIATE`.
Isso coloca um lock exclusivo na tabela no momento da leitura. O segundo pedido 
fica esperando o primeiro terminar. Garantimos atomicidade total.

7. SISTEMA DE TOKENS E LIMITAÇÃO DE USO
================================================================================

7.1. CONCEITO DE JANELA DESLIZANTE (SLIDING WINDOW)

Para evitar que usuários gastem 50k tokens em 1 minuto e fiquem sem uso o resto 
do dia, implementamos uma janela de tempo fixa (ex: 8 horas).
- Ao primeiro uso do dia, `janela_inicio` é marcada.
- Todos os gastos são somados.
- Quando `agora() > janela_inicio + 8 horas`, o contador zera automaticamente 
  no próximo consumo.

7.2. LÓGICA DE CONSUMO ATÔMICO

Fluxo da função `consumir_tokens`:
1. Inicia transação `BEGIN IMMEDIATE`.
2. Lê saldo atual e `janela_inicio`.
3. Verifica se passou de 8h. Se sim, reseta `tokens_usados = 0` e atualiza `janela_inicio`.
4. Calcula `novo_saldo = tokens_usados + amount`.
5. Se `novo_saldo > limite`: Rollback e retorna Erro 402.
6. Se ok: Update `tokens_usados = novo_saldo`.
7. Commit.

7.3. PLANOS: FREE VS. PRO

- Free:
  * Limite: 50.000 tokens por janela.
  * Reset: A cada 8 horas.
  
- Pro:
  * Limite: 500.000 tokens por janela.
  * Reset: A cada 8 horas (ou 24h, configurável).

A mudança de plano é feita via endpoint `/tokens/upgrade`, atualizando o campo 
`plano` e `limite_tokens` na tabela `tokens_usuarios`.

7.4. RESET AUTOMÁTICO DE COTAS

O reset não é feito por um cron job separado (que consumiria recursos). Ele é 
"preguiçoso" (lazy evaluation): ocorre no momento exato em que o usuário tenta 
consumir tokens e o sistema detecta que a janela expirou. Isso economiza ciclos 
de CPU para usuários inativos.

7.5. EXEMPLOS DE CÁLCULO

Cenário: Usuário Free (Limite 50k). Janela iniciou às 08:00. Agora são 10:00.
- Já usou 40.000 tokens.
- Tenta enviar mensagem que custa 5.000.
- Cálculo: 40.000 + 5.000 = 45.000 <= 50.000? SIM.
- Ação: Permite. Novo saldo usado: 45.000.

Cenário: Mesmo usuário, agora são 16:01 (passou de 8h).
- Tenta enviar mensagem de 5.000.
- Sistema detecta: 16:01 > 08:00 + 8h.
- Ação: Reseta `tokens_usados` para 0. Atualiza `janela_inicio` para 16:01.
- Cálculo: 0 + 5.000 <= 50.000? SIM.
- Ação: Permite. Novo saldo usado: 5.000.

10. MANUTENÇÃO E OPERAÇÕES
================================================================================

10.1. BACKUP E RESTAURAÇÃO

Backup:
O banco é UM ÚNICO ARQUIVO. Para fazer backup:
1. Pare o servidor (opcional, mas recomendado para consistência total).
2. Copie `backend/data/cxia.db` para um local seguro (Google Drive, HD Externo).
3. Se quiser backup online automatizado, use um script simples de copy.

Restauração:
1. Pare o servidor.
2. Substitua o arquivo `cxia.db` atual pela cópia de backup.
3. Inicie o servidor.

Dica: O SQLite suporta backup a quente (enquanto roda) usando o comando `.backup` 
no CLI do sqlite3, mas copiar o arquivo com o servidor parado é o método mais 
seguro para leigos.

10.2. LOGS E MONITORAMENTO

O FastAPI/Uvicorn escreve logs no console (stdout).
- Em Linux, redirecione para arquivo: `python main.py >> logs/app.log 2>&1`.
- Monitore erros procurando por "ERROR" ou "CRITICAL".
- Use ferramentas como `htop` para ver uso de CPU/RAM do processo Python.

10.3. OTIMIZAÇÃO DE PERFORMANCE

O sistema já vem com `WAL mode` ativado, que é o maior ganho de performance.
Dicas adicionais:
- Mantenha o arquivo .db em um SSD.
- Se o arquivo crescer muito (>1GB), rode periodicamente: `VACUUM;` (compacta o DB).
- Aumente o `cache_size` do SQLite se tiver RAM sobrando.

10.4. LIMPEZA DE DADOS ANTIGOS

Para evitar que o banco cresça infinitamente:
- Crie um script Python que rode semanalmente deletando mensagens de usuários 
  inativos há mais de 1 ano.
- Exemplo SQL: `DELETE FROM mensagens WHERE criado_em < datetime('now', '-1 year');`

10.5. ATUALIZAÇÃO DE VERSÃO

Para atualizar o backend:
1. Faça backup do `cxia.db`.
2. Pare o serviço.
3. Substitua os arquivos `.py` pelos novos (git pull ou copy).
4. Rode `pip install -r requirements.txt` (caso haja novas libs).
5. Inicie o serviço.
O script `models.py` é idempotente (`IF NOT EXISTS`), então ele não quebrará 
o banco existente ao adicionar novas colunas/tabelas futuras.

11. SOLUÇÃO DE PROBLEMAS (TROUBLESHOOTING)
================================================================================

11.1. ERROS COMUNS DE CONEXÃO

Erro: "Connection Refused"
- Causa: Backend não está rodando ou firewall bloqueando.
- Solução: Verifique se o terminal mostra "Uvicorn running...". Teste `curl http://localhost:8000/health`.

Erro: "CORS Policy" no navegador
- Causa: O frontend está em um domínio não listado no `ALLOWED_ORIGINS`.
- Solução: Adicione o domínio do frontend no `.env` do backend e reinicie.

11.2. PROBLEMAS DE PERMISSÃO DE ARQUIVO

Erro: "unable to open database file"
- Causa: O usuário que roda o Python não tem permissão de escrita na pasta `data/`.
- Solução (Linux): `chmod 777 data/` ou `chown usuario:grupo data/`.
- Solução (Windows): Execute como Administrador ou ajuste propriedades da pasta.

11.3. DEBUG DE JWT INVÁLIDO

Erro: "Invalid token" ou "Token expired"
- Causa: Relógio do servidor ou do cliente dessincronizado, ou `JWT_SECRET` mudou.
- Solução: Verifique se o `JWT_SECRET` no `.env` é o mesmo de quando o token foi gerado. 
  Tokens antigos serão invalidados se a chave mudar (medida de segurança).

11.4. CONFLITOS DE PORTA

Erro: "[Errno 98] Address already in use"
- Causa: Outro programa (ou outra instância do backend) está usando a porta 8000.
- Solução: Mude a porta no `main.py` para 8001 ou mate o processo ocupando a porta.

12. PERGUNTAS FREQUENTES (FAQ)
================================================================================

P: Posso hospedar o backend na Vercel também?
R: Não. A Vercel é serverless e não suporta processos persistentes nem escrita 
   em disco (o arquivo .db seria apagado a cada request). O backend precisa de 
   um servidor VPS (DigitalOcean, Hetzner) ou rodar no seu PC.

P: O SQLite aguenta muitos usuários simultâneos?
R: Sim, para leitura é extremamente rápido. Para escrita, ele trava o arquivo 
   momentaneamente. Para um SaaS com até ~100-200 usuários ativos simultâneos, 
   ele performa muito bem. Acima disso, considere migrar para PostgreSQL (a 
   estrutura do código facilita essa migração futura).

P: E se meu PC desligar?
R: O site ficará indisponível até você ligar o PC novamente. Para produção crítica, 
   recomenda-se hospedar este backend em um VPS barato (ex: $5/mês) que fica 
   ligado 24/7 na nuvem, mas ainda assim sendo "seu" banco privado.

P: Como faço para ver os dados brutos?
R: Baixe o "DB Browser for SQLite". Abra o arquivo `cxia.db`. Você poderá navegar 
   nas tabelas, rodar queries SQL e exportar dados manualmente.

P: Posso mudar o segredo do JWT depois?
R: Sim, mas isso invalidará todas as sessões ativas. Os usuários terão que fazer 
   login novamente.
